  Coding standards may seem insignificant at first, but they are very important for effective software development. They make code more readable for the person who wrote it, making it easier to identify issues and resolve them quickly. They also make it easier for others to read, which is important in collaborative settings.
  Sometimes, we type out code so fast that we don’t even fully understand what we wrote. When bugs emerge in code, the developer should be able to quickly identify the issue and resolve it. If the code is messy and hard to understand, this becomes more difficult. This is why coding standards are important even when code is not intended to be written by others. While it takes slightly more time to write code this way, time will ultimately be saved when debugging. Good coding standards may also cause the developer to write code more deliberately, lowering the chance of making a simple logical mistake.
  When collaborating with others on a project, it is important for team members to write code than is completely mutually intelligible. If everyone is using their own, inconsistent coding standards, the different style will make code difficult to read, and everyone will have a harder time working with each other. That said, coding with others in mind isn’t just necessary for team projects. Even on a solo project, it is possible that the source will have to be shared with others in the future for a variety of reasons. Furthermore, if the software is intended to be open source, many may inspect the code or need to modify it.
